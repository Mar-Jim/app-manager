Local Dev Workbench Tutorial

A practical guide for using this repo to detect projects, generate Databricks Asset Bundle starters, review local command suggestions, create Codex prompts and handoffs, track local work, and draft Azure DevOps updates without committing secrets or surprising teammates.

A. Overview

What It Does

local-dev-workbench is a local Typer CLI, FastAPI backend, and React dashboard for day-to-day developer workflows around Databricks and assistant-assisted coding.

  • Detects local project type from files in the current folder.
  • Generates starter Databricks Asset Bundle projects.
  • Suggests known commands such as bundle validation and tests.
  • Creates structured Codex prompts and handoff/current.md files.
  • Stores todos, worklog entries, local ticket notes, and ADO update drafts in SQLite.

Who It Is For

Developers who already use Git, Python, Databricks, and VS Code, but want a safer local dashboard for repeated setup, validation, prompt, handoff, and ticket-update routines.

It is especially useful in restrictive company environments where secrets, deployment rights, and external writes need careful handling.

Problems It Solves

  • Reduces copy-paste drift in Databricks starter repos.
  • Makes command risk visible before execution.
  • Gives Codex a repeatable, constraint-aware prompt format.
  • Keeps handoff context current between coding sessions.
  • Drafts Azure DevOps updates locally before posting.

What It Does Not Do Yet

  • It does not replace the external deployer repo.
  • It does not generate stg or prd bundle targets.
  • It does not store Azure DevOps PAT values.
  • It does not include an implemented VS Code extension yet; apps/vscode is a placeholder.
  • It does not provide a CLI command to persist ADO config metadata yet.

C. Security Model

Local-First

Source code, detection results, command output, todos, notes, and drafts stay on your machine. The backend binds to 127.0.0.1 by default.

No Secrets in Git

Secrets belong in environment variables or existing local credential stores. Do not put PATs, tokens, hostnames, or customer data in docs, prompts, handoffs, or config committed to Git.

Approval Gates

Generated commands include a risk level. Medium and high risk actions require explicit approval. Destructive command terms are blocked or unsupported.

  • Databricks project repos should define the dev target only.
  • stg and prd promotion belongs to the external deployer repo.
  • Azure DevOps updates are drafted locally first.
  • Posting to Azure DevOps requires explicit --from-draft --yes in the CLI or the web confirmation checkbox.
Do not commit local secrets. Keep .workbench/, .workbench.local.yml, user config files, local SQLite databases, and credential-bearing shell profiles out of commits.

The repo currently reads Azure DevOps settings from environment variables and can store metadata in SQLite. The following YAML layout is the recommended local config shape for repo-local and user-level settings as config-file support grows.

F. Environment Variables

Use environment variables for tokens and machine-specific settings. Do not commit them.

Mac/Linux

export AZURE_DEVOPS_PAT="<YOUR_LOCAL_PAT>"
export WORKBENCH_ADO_ORGANIZATION_URL="https://dev.azure.com/<YOUR_AZURE_DEVOPS_ORG>"
export WORKBENCH_ADO_PROJECT="<YOUR_AZURE_DEVOPS_PROJECT>"
export WORKBENCH_ADO_PAT_ENV_VAR="AZURE_DEVOPS_PAT"
export DATABRICKS_CONFIG_PROFILE="<YOUR_DATABRICKS_PROFILE>"

Windows PowerShell

$env:AZURE_DEVOPS_PAT = "<YOUR_LOCAL_PAT>"
$env:WORKBENCH_ADO_ORGANIZATION_URL = "https://dev.azure.com/<YOUR_AZURE_DEVOPS_ORG>"
$env:WORKBENCH_ADO_PROJECT = "<YOUR_AZURE_DEVOPS_PROJECT>"
$env:WORKBENCH_ADO_PAT_ENV_VAR = "AZURE_DEVOPS_PAT"
$env:DATABRICKS_CONFIG_PROFILE = "<YOUR_DATABRICKS_PROFILE>"
The README documents the default PAT env var name as AZURE_DEVOPS_EXT_PAT. Setting WORKBENCH_ADO_PAT_ENV_VAR="AZURE_DEVOPS_PAT" lets this app read the placeholder requested in this tutorial without changing code.

G. Databricks Setup

The workbench does not replace the Databricks CLI. It suggests and, after approval, runs known Databricks CLI commands against local project folders.

databricks --version
databricks auth profiles
databricks bundle validate -t dev
  • Install the Databricks CLI using your company-approved method.
  • Confirm your local profile exists before validating bundles.
  • Run validation manually once in a generated or existing bundle repo.
  • Use project repos for dev target definitions only.
  • Use the external deployer repo for stg and prd promotion.
The generator enforces --target dev; trying to generate another target raises an error. This is intentional.

workbench create bundle-dashboard my_dashboard

M. Codex Prompt Workflow

The prompt generator creates structured prompts with repo context, constraints, company/security assumptions, Databricks deployment strategy, validation commands, and a handoff update requirement. 

workbench prompt create --task-type add-workflow
workbench prompt create --task-type add-workflow --task "Add a dashboard filter" --save
  1. Run detection in the target repo.
  2. Create a prompt with a specific task description.
  3. Review for secrets or private data before using it.
  4. Save it locally if useful.
  5. Copy the prompt into Codex and keep validation commands visible.
Prompts intentionally include constraints such as dev-only bundle targets, external deployer ownership for stg and prd, and approval before deployment or cloud-mutating commands.

N. Handoff Workflow

handoff/current.md keeps context fresh between Codex chats. Create or update it when a task changes direction, before ending a session, after meaningful files change, or before handing work to a teammate. 

workbench handoff create
workbench handoff show

A good cadence is: update handoff after implementation, after validation, and any time there is a blocker or assumption the next session must know.

O. Todos and Worklog

Todos and worklog entries are stored locally in .workbench/workbench.sqlite3 for the current working directory.

workbench todo add "Validate bundle generator"
workbench todo list
workbench todo complete 1

workbench worklog add "Tested Databricks CLI auth"
workbench worklog summary

The daily summary is a local draft with sections for work completed, blockers, and next steps. It is not posted anywhere automatically.

P. Azure DevOps Workflow

Azure DevOps support is permissioned and local-first. Ticket reads use the Azure DevOps REST API when configured. Local notes and draft updates are stored in SQLite.

Configure Org, Project, and PAT Env Var

export WORKBENCH_ADO_ORGANIZATION_URL="https://dev.azure.com/<YOUR_AZURE_DEVOPS_ORG>"
export WORKBENCH_ADO_PROJECT="<YOUR_AZURE_DEVOPS_PROJECT>"
export WORKBENCH_ADO_PAT_ENV_VAR="AZURE_DEVOPS_PAT"
export AZURE_DEVOPS_PAT="<YOUR_LOCAL_PAT>"

Implemented Commands

workbench ado tickets list
workbench ado ticket show <TICKET_ID>
workbench ado ticket draft-update <TICKET_ID> --note "Validated locally"
workbench ado ticket post-update <TICKET_ID> --from-draft --yes
Planned command: workbench ado config check is not implemented yet. For now, use workbench ado tickets list; if config or token values are missing, it prints setup guidance instead of crashing.

Posting should remain optional. Draft first, review the draft, then post only when you have explicit approval to update the ticket.

R. Troubleshooting

workbench Not Found

Activate .venv and reinstall with python -m pip install -e ".[dev]".

Python Version Mismatch

Use Python 3.11 or newer. Recreate the virtual environment with the correct interpreter.

npm install Fails

Use Node.js 20+ and the company-approved npm registry or proxy configuration.

Backend Port in Use

Run workbench serve --port 8788 and adjust the Vite proxy if needed.

Databricks CLI Not Authenticated

Run databricks auth profiles and fix your local profile before using bundle commands.

ADO PAT Missing

Set the PAT env var named by WORKBENCH_ADO_PAT_ENV_VAR. Do not store the token in SQLite or Git.

Detection Says Unknown

Run from the project root and check for expected markers such as databricks.yml, app.yaml, or pyproject.toml.

Bundle Validate Fails

Run the Databricks CLI command manually, inspect YAML/resource paths, and confirm you are targeting dev.

Local Config Accidentally Committed

Remove secrets from Git history using your approved company process, rotate exposed credentials, and add the path to .gitignore.

S. Team Onboarding Guide

  1. Clone the repo from <YOUR_REPO_URL>.
  2. Create a Python 3.11+ virtual environment and install with python -m pip install -e ".[dev]".
  3. Run npm install.
  4. Configure local environment variables for Databricks and Azure DevOps if needed.
  5. Run workbench doctor and pytest.
  6. Confirm Databricks CLI auth with databricks auth profiles.
  7. Start workbench serve and npm run web:dev.
  8. Open the dashboard and run project detection.
  9. Create the first handoff with workbench handoff create.
  10. Before every commit, check that secrets, local config, SQLite files, and private output are not staged.

T. FAQ

Does this deploy to production?

No. Project repos are dev-focused. Production promotion belongs to the external deployer repo.

Does this replace the deployer repo?

No. It complements it by helping developers validate and organize local dev work.

Does it send code to AI?

No. It generates local prompt text. You decide what to copy into Codex after reviewing it.

Where are todos stored?

In local SQLite at .workbench/workbench.sqlite3 under the current working directory.

Can I use it without Azure DevOps?

Yes. The rest of the CLI and dashboard continue to work. The Tickets page shows setup guidance when ADO is not configured.

Can I use it without VS Code?

Yes. VS Code integration is optional and future-facing. The current app runs from the CLI and browser.

Why only dev target in project repos?

It keeps local developer repos simple and avoids duplicating staging/production promotion logic owned by the external deployer repo.

Can I customize templates?

Yes. Templates live under templates/databricks_bundle_*. Keep changes dev-only unless the deployment model changes.

Standalone file: no external fonts, images, scripts, CDNs, or JavaScript libraries required.
